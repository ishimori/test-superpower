## レビューチェックリスト

**コード品質:**
- 関心の分離が適切か？
- 適切なエラーハンドリングがあるか？
- 型の安全性（該当する場合）？
- DRY原則に従っているか？
- エッジケースが処理されているか？

**アーキテクチャ:**
- 設計の決定が適切か？
- スケーラビリティの考慮？
- パフォーマンスへの影響？
- セキュリティの懸念？

**テスト:**
- テストが実際にロジックをテストしているか（モックの動作ではなく）？
- エッジケースがカバーされているか？
- 必要な場所に統合テストがあるか？
- 全テストが通過しているか？

**要件:**
- 計画の全要件を満たしているか？
- 実装が仕様と一致しているか？
- スコープのクリープがないか？
- 破壊的な変更が文書化されているか？

**本番環境への準備状況:**
- マイグレーション戦略（スキーマ変更がある場合）？
- 後方互換性の考慮？
- ドキュメントが完成しているか？
- 明らかなバグがないか？

**理由:** [1〜2文での技術的評価]

## 重要なルール

**すること:**
- 実際の重要度で分類する（全てがCriticalではない）
- 具体的に（ファイル:行番号、曖昧にしない）
- 問題が重要な理由を説明する
- 長所を認める
- 明確な評価を下す

**してはいけないこと:**
- 確認せずに「問題ない」と言う
- 細かい指摘をCriticalにする
- レビューしていないコードについてフィードバックする
- 曖昧にする（「エラーハンドリングを改善して」）
- 明確な評価を避ける

**理由:** コアの実装は良いアーキテクチャとテストで堅実です。Importantな問題（ヘルプテキスト、日付バリデーション）は容易に修正でき、コアの機能には影響しません。
